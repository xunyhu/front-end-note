# 面试题：项目如何做环境区分？（标准回答）

在前端项目中，环境区分的核心目标是：**让不同环境（开发、测试、生产）可以使用不同的配置，而不影响代码逻辑本身**。

---

### 一、常见环境类型

一般项目会分为三种环境：

* **开发环境（development）**：本地开发使用
* **测试环境（test / staging）**：联调或测试使用
* **生产环境（production）**：线上正式环境

---

### 二、环境变量配置方式

在 React（如 Vite / Webpack）项目中，通常通过环境变量文件来区分：

* `.env.development`
* `.env.test`
* `.env.production`

例如：

```bash
VITE_API_BASE_URL=https://dev-api.xxx.com
```

生产环境则是：

```bash
VITE_API_BASE_URL=https://api.xxx.com
```

---

### 三、代码中的使用方式

在项目中通过 `import.meta.env` 或 `process.env` 读取环境变量：

```js
const baseURL = import.meta.env.VITE_API_BASE_URL
```

然后在请求封装中统一使用：

```js
axios.create({
  baseURL
})
```

---

### 四、接口环境隔离

在实际项目中，接口一般会做统一管理：

* 开发环境：本地 mock 或 dev 服务器
* 测试环境：联调服务器
* 生产环境：正式 API

通过环境变量切换 baseURL，实现无侵入切换。

---

### 五、小程序项目中的区别

在小程序（Taro / 自研框架）中也类似：

* 使用不同的 build 配置
* 或通过不同的 env 文件控制请求域名

例如：

* dev：api.dev.xxx.com
* prod：api.xxx.com

---

### 六、部署层面的环境区分

除了前端代码，还会在部署层做区分：

* CI/CD 自动注入环境变量
* 不同分支对应不同环境（dev / release / master）
* Nginx 或网关层做域名隔离

---

### 七、实际项目中的做法

在我的项目中：

* 使用 `.env` 文件区分开发和生产环境
* 前端统一通过 `VITE_API_BASE_URL` 控制接口域名
* 线上环境通过 CI/CD 自动注入配置
* 小程序和 Web 保持同一套环境配置逻辑

---

### 八、总结

环境区分的核心原则是：

> **配置与代码解耦，通过环境变量控制不同运行环境的行为**

这样可以做到：

* 一套代码，多环境运行
* 降低人为切换错误
* 提高部署和维护效率
