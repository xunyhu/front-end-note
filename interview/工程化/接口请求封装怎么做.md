# 面试题：接口请求封装怎么做？（标准回答）

在前端项目中，接口请求封装的核心目标是：**统一请求规范、降低重复代码、提升可维护性，并增强错误处理与扩展能力**。

---

### 一、基础封装思路

一般会基于 Axios（或 fetch）进行二次封装，核心包括：

* 统一 baseURL 配置
* 统一超时设置
* 请求 / 响应拦截器
* 错误统一处理

---

### 二、请求实例封装

通常会先创建一个 request 实例：

```js
import axios from 'axios'

const request = axios.create({
  baseURL: import.meta.env.VITE_API_BASE_URL,
  timeout: 10000
})
```

---

### 三、请求拦截器

主要做两件事：

#### 1. 自动携带 token

```js
request.interceptors.request.use(config => {
  const token = localStorage.getItem('token')
  if (token) {
    config.headers.Authorization = `Bearer ${token}`
  }
  return config
})
```

#### 2. 统一参数处理

比如加请求时间戳、防缓存等

---

### 四、响应拦截器

统一处理后端返回结构：

```js
request.interceptors.response.use(
  response => {
    const res = response.data
    if (res.code !== 200) {
      return Promise.reject(res.message)
    }
    return res.data
  },
  error => {
    return Promise.reject(error)
  }
)
```

---

### 五、统一错误处理

在响应层做统一处理：

* 401：未登录 → 跳转登录页
* 403：无权限
* 500：服务器错误提示

---

### 六、API 层拆分

为了结构清晰，一般会按模块拆分 API：

```
/api
  ├── user.ts
  ├── order.ts
  ├── product.ts
```

例如：

```js
export const getUserInfo = () => {
  return request.get('/user/info')
}
```

---

### 七、进阶优化（项目经验）

在实际项目中，还会做一些增强能力：

#### 1. 请求防重复

* 相同请求拦截（pending request cache）

#### 2. Loading 统一管理

* 自动控制全局 loading 或局部 loading

#### 3. 重试机制

* 请求失败自动重试（可选）

#### 4. 类型约束（TypeScript）

* 对接口返回值做泛型封装

---

### 八、在小程序 / Taro 中的差异

在 Taro 或小程序中：

* 使用 Taro.request 或封装 fetch
* 同样做统一拦截
* 注意兼容不同端（H5 / 小程序）

---

### 九、实际项目中的做法

在我的项目中：

* 使用 Axios 封装 request 层
* 统一处理 token 注入和 401 跳转
* 按业务模块拆分 API 文件
* 在零售小程序中增加了请求合并与防重复机制

---

### 十、总结

接口封装的核心是：

> **统一入口 + 统一处理 + 业务解耦**

通过这一层封装，可以让业务层只关心“发请求”，而不用关心“怎么请求”。
