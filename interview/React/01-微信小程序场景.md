## 一、项目深挖

1. 你做过的小程序项目是怎样的架构？

    我之前做过两个类型的小程序项目，一个是面向酒店的订房订餐小程序，另一个是山姆会员商店零售小程序。

    其中酒店小程序主要技术栈是 **Taro + React + MobX**，而山姆零售小程序是基于**自研框架 + React + Redux** 来实现的，两套体系在架构设计上思路是一致的，只是状态管理和工程封装方式有所不同。

    整体架构我一般分为四层：

    第一层是**页面层（pages）**，主要负责页面结构和路由，对应小程序的页面入口，比如首页、商品页、购物车、订单页等。

    第二层是**业务组件层（components / business components）**，比如商品卡片、SKU选择弹窗、购物车组件、订单列表等，这一层主要做业务级复用封装。

    第三层是**状态管理层**，酒店项目使用的是 MobX，山姆项目使用 Redux，主要负责管理全局状态，比如用户信息、购物车、门店信息、会员状态等。

    第四层是**数据请求层（services）**，统一封装接口请求，包括 token 注入、错误处理、loading 状态、接口拦截等逻辑。

    在工程能力上，尤其是在山姆项目中，我们基于自研框架做了一些增强能力，比如：

    * 统一 request 封装（解决多端差异和接口标准化）
    * hooks 复用（如 useCart、useUser、useMember）
    * 业务组件库沉淀（商品卡片、价格模块、营销组件等）

    在小程序特性适配方面也做了一些优化，例如：

    * 分包加载优化首屏性能
    * 页面数据缓存与局部更新机制
    * 利用小程序生命周期做数据刷新和状态同步

    整体目标是让业务开发尽可能标准化和组件化，实现“配置驱动 + 复用优先”的开发模式，提高多人协作效率和开发一致性。

2. 餐饮/零售业务核心流程怎么实现？

    餐饮/零售核心链路其实很固定：商品 → 购物车 → 下单 → 支付 → 订单状态

    我在做餐饮和零售类小程序时，整体业务流程本质是围绕“商品购买链路”来设计的，一般可以拆成几个核心环节：

    首先是商品浏览阶段，用户进入首页或者分类页，通过接口获取商品列表数据，这一层会结合门店、用户身份以及会员信息做一些差异化展示，比如价格、库存、促销活动等。

    第二步是商品选择与加入购物车，在这个阶段会涉及 SKU 选择、规格组合、加购数量控制等逻辑。购物车数据一般会做本地缓存 + 全局状态管理，确保多页面之间数据一致，比如 MobX 或 Redux 来统一管理购物车状态。

    第三步是下单阶段，用户从购物车进入结算页，会进行订单生成。这里会做一些关键校验，比如库存校验、价格实时校验、优惠券计算等，同时会防止重复提交，通过按钮 loading 或接口幂等来处理。

    第四步是支付流程，调用微信支付能力，由后端返回支付参数，前端调起支付 API，支付成功后通过回调或者轮询方式更新订单状态。

    最后是订单状态管理，包括待支付、已支付、已取消、配送中等状态，这部分通常会和后端订单系统同步，同时前端会做轮询或者 WebSocket 方式来更新状态。

3. 你做过最复杂的页面/功能是什么？

    我在项目中做过相对比较复杂的页面主要有两个，一个是零售类的商品详情 + SKU + 加购链路页面，另一个是购物车结算页，这两个页面在业务复杂度上是比较高的。

    以商品详情页为例，这个页面不仅仅是展示商品信息，还涉及到 SKU 多规格组合选择、库存实时校验、价格联动计算、促销活动展示（比如满减、折扣、会员价等），同时还要支持加入购物车和立即购买两种不同路径。

    在实现上，SKU 选择是一个状态驱动模型，当用户选择不同规格时，需要动态计算当前 SKU 是否可用、对应价格以及库存状态，同时要处理不同组合之间的联动关系，避免用户选到无效组合。

    购物车页面相对更复杂一些，需要处理多商品、多门店、多规格的数据结构，同时涉及到价格实时计算，比如优惠券、满减活动、会员折扣叠加计算等逻辑。

    在状态管理上，我一般会使用 Redux 或 MobX 来做全局状态维护，同时配合本地缓存做持久化，确保页面刷新或者切换不会丢失购物车数据。

    另外在性能和体验上，也做了一些优化，比如 SKU 弹窗的局部渲染优化、购物车批量更新防抖处理，以及接口层的合并请求，避免频繁触发重新渲染。

    整体来说，这类页面的*难点不在 UI，而是在于复杂状态管理 + 多规则计算 + 数据一致性保证*。

4. 小程序性能优化怎么做？

    在小程序项目中，我做性能优化主要是从四个层面来考虑：首屏加载、渲染性能、接口性能以及包体积优化。

    首先是首屏加载优化，主要通过分包加载来减少主包体积（主/分包大小 ≤ 2 MB， 总包大小不超过20MB），把非核心页面（比如订单详情、个人中心、帮助页等）拆到分包中，确保用户进入首页时只加载核心业务资源，从而提升首屏打开速度。

    第二是渲染性能优化，在页面层面会尽量避免不必要的 setState 更新，控制数据粒度，减少大对象频繁修改。同时对于列表类页面，会做分页加载或者虚拟列表处理，避免一次性渲染大量节点。

    第三是接口性能优化，主要包括接口合并、请求缓存以及防抖处理，比如购物车更新、搜索接口等会做 debounce 或节流处理，同时对部分高频数据做本地缓存，减少重复请求。

    第四是包体积优化，主要通过按需引入组件和工具库，避免全量引入第三方依赖，同时对静态资源（图片、素材）进行压缩和 CDN 化处理。

    在我做的项目里（比如山姆零售小程序），还会结合业务做一些优化，比如购物车数据本地缓存 + 增量更新，避免每次都全量刷新页面状态，同时利用小程序生命周期（onShow）做局部刷新，而不是整页重载。

    整体优化思路是：减少首屏负担 + 减少无效渲染 + 减少网络请求 + 控制包体积。

## 二、React 技术考察

1. React Hooks 原理 & 常见问题

    React Hooks 的本质是让函数组件具备状态和生命周期能力，它的底层依赖是 React 的“链表结构 + 按顺序调用机制”。React 会在组件内部维护一个 hooks 链表，通过调用顺序来对应每一个 hook 的状态。

    * Hooks 的基本原理

        在函数组件每次渲染时，React 会重新执行整个函数，但 Hooks 的状态不会丢失，是因为 React 在内部通过 Fiber 节点保存了 hooks 的状态链表。
        每个 hook（如 useState、useEffect）都会被按顺序挂载到当前 fiber 上，下次渲染时通过“调用顺序”去复用之前的状态。
        因此 Hooks 有一个非常重要的规则：不能在条件语句或循环中调用 Hooks，否则会导致调用顺序错乱。

    * useState 原理

        useState 本质是一个闭包 + 链表结构，每次 setState 实际上是触发一次更新调度（schedule update），然后重新执行函数组件。
        React 会将新的 state 存入对应 fiber 节点中，并触发 re-render。

    * useEffect 原理

        useEffect 会在 render 阶段收集依赖，在 commit 阶段异步执行。
        它的执行时机是在 DOM 更新之后，不会阻塞渲染。
        依赖数组的变化会触发 effect 的重新执行，否则会复用上一次的 effect。

    * 常见问题

        * useEffect 无限循环

            通常是因为依赖数组写错，比如依赖了一个每次 render 都变化的对象或函数。

        * 闭包问题

            Hooks 中的函数会“捕获旧值”，导致数据不是最新的，需要通过 useRef 或依赖更新解决。

        * 性能问题

            频繁 setState 会导致组件重复渲染，可以通过 useMemo、useCallback、React.memo 做优化。

        * 为什么不能条件调用 Hooks

            因为 Hooks 依赖调用顺序来维护状态，如果顺序变化会导致状态错乱。

    Hooks 的核心本质是： 通过 Fiber + 顺序链表，在函数组件中“模拟状态和生命周期”。
